## Clarifications

### Session 2026-03-06

- Q: 시민이 예약 시 세부 타임(00/20/40분)을 직접 선택하는가? → A: 시민이 세부 타임(예: 4시 20분)을 직접 선택한다.
- Q: 예약 완료 시 별도 확인코드를 부여하는가? → A: 이름+전화번호만으로 식별한다 (별도 코드 없음).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 세금상담 예약 생성 (Priority: P1)

시민이 QR코드를 스캔하여 모바일 웹으로 접속한 뒤, 개인정보 활용에 동의하고
원하는 시간대와 상담 분야를 선택하여 세금상담 예약을 완료한다.

**Why this priority**: 시스템의 핵심 가치. 예약 생성 없이는 다른 모든 기능이
의미 없다. 이 스토리만으로도 MVP로 운영 가능하다.

**Independent Test**: QR코드(또는 URL 직접 접속)로 예약 페이지에 진입하여
시간대 선택 → 정보 입력 → 예약 완료까지의 전체 흐름을 테스트할 수 있다.

**Acceptance Scenarios**:

1. **Given** 예약 가능한 슬롯이 남아있을 때, **When** 시민이 QR코드를
   스캔하면, **Then** 예약 페이지가 즉시 표시된다.
2. **Given** 예약 페이지에 도달했을 때, **When** 시민이 개인정보 활용동의에
   체크하지 않으면, **Then** 다음 단계로 진행할 수 없다.
3. **Given** 개인정보 동의를 완료했을 때, **When** 시민이 시간대(4시/5시),
   세부 타임(00분/20분/40분), 상담 분야(국세/지방세/기타)를 선택하면,
   **Then** 이름과 전화번호 입력 화면이 표시된다.
4. **Given** 유효한 이름과 전화번호를 입력했을 때, **When** 시민이 예약
   확인 버튼을 누르면, **Then** 예약이 저장되고 완료 화면이 표시된다.
5. **Given** 특정 시간대의 예약이 18건(만석)일 때, **When** 시민이 해당
   시간대를 선택하려 하면, **Then** 만석 안내 메시지가 표시되고 선택이
   불가능하다.
6. **Given** 특정 세부 타임(예: 4시 20분)의 예약이 6건(만석)일 때,
   **When** 시민이 해당 타임을 선택하려 하면, **Then** 만석 안내가
   표시되고 선택이 불가능하다.
7. **Given** 두 명의 시민이 마지막 1자리 남은 슬롯에 동시 예약을 시도할 때,
   **When** 둘 다 제출하면, **Then** 한 명만 성공하고 다른 한 명에게는
   만석 안내가 표시된다.
8. **Given** 개인정보 동의 화면에서, **When** 동의 내용이 표시될 때,
   **Then** "개인정보는 수집일로부터 1주일 이내 폐기됩니다" 문구가
   반드시 포함되어 있다.

### User Story 2 - SMS 예약 확인 및 알림 (Priority: P2)

예약을 완료한 시민에게 확인 문자가 즉시 발송되고, 상담 시간 5분 전에
알림 문자가 자동 발송된다.

**Why this priority**: 예약 확인 수단이 없으면 시민이 자신의 예약 시간을
기억하기 어렵고, 불참률이 높아진다. 예약 신뢰도의 핵심이다.

**Independent Test**: 예약 생성 후 확인 문자 수신 여부를 확인하고,
상담 시간 5분 전 알림 문자가 발송되는지 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 시민이 예약을 완료했을 때, **When** 예약이 저장되면,
   **Then** 입력한 전화번호로 예약 확인 문자가 즉시 발송된다.
2. **Given** 확인 문자 내용에, **When** 문자가 발송되면, **Then**
   예약자 이름, 상담 시간, 상담 분야, 장소(강남구민회관) 정보가
   포함되어 있다.
3. **Given** 예약된 상담 시간 5분 전이 되었을 때, **When** 시스템이
   자동으로, **Then** 해당 시민에게 알림 문자를 발송한다.
4. **Given** 문자 발송이 실패했을 때, **When** 발송 오류가 발생하면,
   **Then** 시스템이 재시도하거나 실패 내역을 기록한다.

---

### User Story 3 - 예약 취소 (Priority: P3)

예약한 시민이 자신의 예약을 조회하고 취소할 수 있다. 취소 시 해당 슬롯이
다시 개방되고, 취소 확인 문자가 발송된다.

**Why this priority**: 시민이 사정이 생겨 예약을 취소할 수 있어야 다른
시민에게 기회가 돌아간다. 예약 무결성과 직결된다.

**Independent Test**: 기존 예약을 이름+전화번호로 조회한 뒤 취소 버튼을
눌러 슬롯이 다시 개방되는지, 취소 확인 문자가 발송되는지 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 시민이 예약 취소 페이지에 접속했을 때, **When** 예약 시
   입력한 이름과 전화번호를 입력하면, **Then** 해당 예약 정보가 조회된다.
2. **Given** 예약 정보가 조회되었을 때, **When** 시민이 취소 버튼을
   누르면, **Then** 예약이 취소되고 해당 슬롯이 다시 예약 가능해진다.
3. **Given** 예약이 취소되었을 때, **When** 취소가 완료되면, **Then**
   시민에게 취소 확인 문자가 발송된다.
4. **Given** 존재하지 않는 이름/전화번호로 조회했을 때, **When** 검색
   결과가 없으면, **Then** 예약을 찾을 수 없다는 안내가 표시된다.

### User Story 4 - 관리자 예약 현황 관리 (Priority: P4)

관리자가 비밀번호로 로그인하여 전체 예약 현황을 조회하고,
필요 시 특정 예약을 수동으로 취소할 수 있다.

**Why this priority**: 현장 운영 시 관리자가 예약 상황을 파악하고
노쇼 또는 문제 예약을 처리해야 한다. 운영 필수 기능이나
시민 예약 기능 이후에 구현해도 충분하다.

**Independent Test**: 관리자 비밀번호로 로그인 후 예약 목록 조회,
시간대별 예약 건수 확인, 특정 예약 수동 취소를 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 관리자 페이지에 접속했을 때, **When** 올바른 비밀번호를
   입력하면, **Then** 예약 현황 화면에 진입할 수 있다.
2. **Given** 잘못된 비밀번호를 입력했을 때, **When** 로그인을 시도하면,
   **Then** 접근이 거부된다.
3. **Given** 예약 현황 화면에서, **When** 관리자가 목록을 조회하면,
   **Then** 시간대별 예약 목록과 잔여 슬롯 수가 표시된다.
4. **Given** 특정 예약을 선택했을 때, **When** 관리자가 취소 버튼을
   누르면, **Then** 해당 예약이 취소되고 슬롯이 개방된다.

---

### User Story 5 - 개인정보 폐기 (Priority: P5)

이벤트 종료 후 관리자가 수집된 개인정보를 폐기하거나, 수집 후
7일이 경과하면 시스템이 자동으로 삭제한다.

**Why this priority**: 법적 요건이므로 반드시 구현해야 하나,
이벤트(3/12) 이후에 실행되므로 구현 우선순위는 낮다.

**Independent Test**: 관리자 화면에서 개인정보 폐기 버튼을 눌러
데이터 삭제를 검증하거나, 7일 경과 후 자동 삭제를 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 이벤트가 종료된 후, **When** 관리자가 개인정보 폐기
   버튼을 누르면, **Then** 모든 예약자 개인정보(이름, 전화번호)가
   복구 불가능하게 삭제된다.
2. **Given** 개인정보 수집일로부터 7일이 경과했을 때, **When**
   시스템이 자동 점검하면, **Then** 해당 개인정보가 자동 삭제된다.
3. **Given** 개인정보 폐기가 완료되었을 때, **When** 관리자가
   예약 목록을 조회하면, **Then** 개인정보가 표시되지 않는다.

---

### Edge Cases

- 모든 시간대(36건)가 만석일 때 예약 페이지에 접속하면 어떻게 되는가?
  → 전체 만석 안내 메시지를 표시하고 예약 폼을 비활성화한다.
- 같은 이름+전화번호로 중복 예약을 시도하면 어떻게 되는가?
  → 이미 예약이 존재한다는 안내와 함께 기존 예약 정보를 표시한다.
- 이벤트 날짜(3/12) 이후에 예약 페이지에 접속하면 어떻게 되는가?
  → 이벤트 종료 안내를 표시하고 예약 기능을 비활성화한다.
- 전화번호를 잘못된 형식으로 입력하면 어떻게 되는가?
  → 유효한 휴대전화 번호 형식(010-XXXX-XXXX)인지 검증하여 오류 안내한다.
- 예약 확인 문자 발송 중 네트워크 오류가 발생하면 어떻게 되는가?
  → 예약 자체는 성공 처리하고, 문자 발송을 재시도한다. 최종 실패 시 로그를 기록한다.
- 관리자가 이미 취소된 예약을 다시 취소하려 하면 어떻게 되는가?
  → 이미 취소된 예약이라는 안내를 표시한다.
- 특정 세부 타임만 만석이고 같은 시간대의 다른 타임은 여유가 있을 때?
  → 만석인 타임만 선택 불가로 표시하고, 여유 있는 타임은 선택 가능하다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 QR코드 스캔(또는 URL 직접 접속)으로 예약 페이지를
  즉시 표시해야 한다.
- **FR-002**: 시스템은 예약 전 개인정보 활용동의를 필수로 받아야 하며,
  동의 화면에 "개인정보는 수집일로부터 1주일 이내 폐기됩니다" 문구를
  포함해야 한다.
- **FR-003**: 시스템은 시간대(오후 4시, 오후 5시), 세부 타임(00분, 20분,
  40분), 상담 분야(국세, 지방세, 기타) 선택 기능을 제공해야 한다.
- **FR-004**: 시스템은 예약자 이름(필수)과 전화번호(필수)만 수집해야 한다.
- **FR-005**: 시스템은 세부 타임당 최대 6건, 시간대별 최대 18건, 총 36건의
  예약 상한을 초과하지 않도록 강제해야 한다.
- **FR-006**: 시스템은 동시 접속 시 초과 예약이 발생하지 않도록 동시성
  제어를 해야 한다.
- **FR-007**: 시스템은 만석인 시간대 및 세부 타임을 선택 불가 상태로
  표시해야 한다.
- **FR-008**: 시스템은 예약 완료 시 확인 문자를 즉시 발송해야 한다.
- **FR-009**: 시스템은 상담 시간 5분 전 알림 문자를 자동 발송해야 한다.
- **FR-010**: 시스템은 이름+전화번호로 기존 예약을 조회하는 기능을
  제공해야 한다.
- **FR-011**: 시스템은 시민이 자신의 예약을 취소할 수 있는 기능을
  제공하고, 취소 시 해당 슬롯을 다시 개방해야 한다.
- **FR-012**: 시스템은 예약 취소 시 취소 확인 문자를 발송해야 한다.
- **FR-013**: 시스템은 관리자가 비밀번호로 로그인하여 전체 예약 현황을
  조회할 수 있는 화면을 제공해야 한다.
- **FR-014**: 시스템은 관리자가 특정 예약을 수동으로 취소할 수 있어야 한다.
- **FR-015**: 시스템은 수집 후 7일이 경과한 개인정보를 자동 삭제하거나,
  관리자가 수동으로 폐기할 수 있는 기능을 제공해야 한다.
- **FR-016**: 시스템은 같은 이름+전화번호로 중복 예약을 방지해야 한다.
- **FR-017**: 시스템은 문자 발송 실패 시 재시도하거나 실패 내역을
  기록해야 한다.
- **FR-018**: 시스템은 모든 화면을 모바일 뷰포트에 최적화해야 한다.
- **FR-019**: 시스템은 이벤트 종료(2026-03-12) 이후 예약 기능을
  비활성화해야 한다.
- **FR-020**: 시스템은 전화번호 형식(휴대전화)을 검증해야 한다.

### Key Entities

- **예약(Reservation)**: 시민의 세금상담 예약. 예약자 이름, 전화번호,
  시간대, 세부 타임, 상담 분야, 예약 상태(활성/취소), 생성 시각을 포함한다.
- **시간대 슬롯(Time Slot)**: 상담 가능 시간 단위. 시간대(4시/5시),
  세부 타임(00분/20분/40분), 현재 예약 건수, 최대 수용 건수(6건/타임)
  정보를 포함한다.
- **SMS 발송 기록(SMS Log)**: 문자 발송 이력. 수신자 전화번호,
  발송 유형(확인/알림/취소), 발송 상태(성공/실패/재시도), 발송 시각을
  포함한다.

## Assumptions

- QR코드는 예약 페이지 URL을 직접 가리키며, 별도의 랜딩 페이지 없이
  즉시 예약 화면으로 진입한다.
- 전화번호 형식은 한국 휴대전화(010-XXXX-XXXX)로 한정한다.
- 중복 예약 판단 기준은 이름+전화번호 조합의 일치로 한다.
- 관리자는 1명(또는 소수)이며, 동시 관리자 접속에 대한 별도 처리는
  불필요하다.
- SMS 발송 재시도는 최대 3회로 제한한다.
- 이벤트 종료일(3/12) 자정 이후 예약 기능을 자동 비활성화한다.
- 개인정보 수집 목적은 "통계 및 예약정보 제공"으로 고정한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 시민이 QR코드 스캔부터 예약 완료까지 2분 이내에 완료할
  수 있다.
- **SC-002**: 동시 접속자 50명이 동시에 예약을 시도해도 초과 예약 없이
  정상 처리된다.
- **SC-003**: 예약 완료 후 30초 이내에 확인 문자가 발송된다.
- **SC-004**: 시민의 90% 이상이 첫 시도에 예약을 성공적으로 완료한다.
- **SC-005**: 만석 시간대의 예약 차단율이 100%이다 (초과 예약 0건).
- **SC-006**: 총 36건의 상담 예약이 정상적으로 관리된다.
- **SC-007**: 이벤트 종료 후 7일 이내에 모든 개인정보가 폐기된다.
